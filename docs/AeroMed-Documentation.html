1. Executive Summary

AeroMed is a production-grade DevOps platform engineered for a global air ambulance network providing critical-care transport, emergency medical evacuation, and disaster-response support.

In this domain, infrastructure failure is a patient-safety event: a service outage mid-evacuation can sever flight telemetry or block access to a patient's medical record during critical care. The platform is therefore architected around three non-negotiable principles — high availability, observability, and provable disaster recovery.

The solution delivers six independently deployable microservices behind a single API gateway, containerized with Docker, orchestrated on Kubernetes (AWS EKS), provisioned through Terraform, delivered via Jenkins CI/CD, and observed end-to-end with Prometheus, Grafana, and AlertManager. A complete disaster-recovery programme — six tested runbooks with tiered RTO/RPO objectives and scripted backup/restore/failover — completes the platform.

2. Problem Statement & Objectives

2.1 Business Context

AeroMed Emergency Services coordinates aircraft operations, patient records, medical-equipment monitoring, flight planning, and emergency-response workflows across multiple regions. Existing systems suffered from deployment inconsistency, limited monitoring, and poor disaster-recovery readiness.

2.2 Objective

Design and implement a highly available DevOps ecosystem supporting emergency healthcare operations and infrastructure resilience — capable of withstanding simulated aircraft-communication failures, medical-system outages, infrastructure disruptions, and emergency traffic surges.

3. System Architecture

AeroMed follows a microservices architecture with strict fault isolation. All external traffic enters through a single internet-facing API Gateway; the five domain services are never exposed directly. Services communicate over HTTP with explicit timeouts so that a slow or failed dependency degrades gracefully instead of cascading. Every service is stateless and independently scalable.

Figure 1 — Logical system architecture (Tier 1 services in red, Tier 2 in amber)

3.1 Design Principles

  • Fault isolation — a failure in one service cannot bring down another.
  • Single entry point — only the API Gateway is public; everything else is internal-only.
  • Stateless services — any pod can be rescheduled to any node without data loss.
  • Observability by default — every service exposes /health and /metrics.
  • Tiered criticality — services are graded Tier 1/2/3, which drives recovery objectives.

5.1 Representative Endpoint (aircraft-comms)

@app.route("/health")          # Kubernetes liveness/readiness probe target
def health():
    return jsonify({"status": "healthy", "service": SERVICE_NAME, "version": "1.0.0"})

@app.route("/metrics")         # Prometheus scrape endpoint
def metrics():
    return generate_latest(), 200, {"Content-Type": CONTENT_TYPE_LATEST}

@app.route("/api/status")      # Business data — aircraft telemetry
def api_status():
    return jsonify({"aircraft_tracked": len(MOCK_COMMS), "communications": MOCK_COMMS})
API Gateway pattern: the gateway's /api/all-status fans out to all five backends and returns one aggregated health view — the single source of truth used by dashboards and the recovery demo.

6. Containerization (Docker)

Each service ships as an immutable Docker image built from a hardened, production-oriented Dockerfile.

FROM python:3.11-slim                 # minimal base image
RUN useradd appuser                     # non-root user (security)
COPY requirements.txt .                 # deps first -> layer caching
RUN pip install --no-cache-dir -r requirements.txt
COPY app.py .
USER appuser                            # drop root privileges
HEALTHCHECK CMD ... /health             # container-level health
CMD ["gunicorn","--bind","0.0.0.0:5005","--workers","2","--threads","2","app:app"]

Engineering decisions

  • Slim base image reduces attack surface and image size.
  • Layer caching — copying requirements.txt before code avoids re-installing dependencies on every code change.
  • Non-root execution — containers never run as root.
  • Gunicorn — a production WSGI server (multiple workers/threads), not the Flask dev server.
Deployment evidence: all ten containers (6 services + Prometheus + Grafana + AlertManager + Jenkins) build and run successfully via Docker Compose, exposed on ports 5050, 5001-5005, 9090, 3000, 9093, and 8080.

8. CI/CD Pipeline (Jenkins)

Delivery is automated with Jenkins. A master pipeline plus one pipeline per service share a reusable pipeline library exposing three steps — aeromedBuild, aeromedDeploy, and aeromedNotify — so build, deploy, and notification logic live in a single versioned place.

Figure 3 — CI/CD pipeline flow

Pipeline Stages

  1. Checkout — pull source, capture git commit.
  2. Pre-flight — verify Docker & cluster reachability.
  3. Build — aeromedBuild.buildImage() builds and dual-tags the image (build number + latest).
  4. Security Scan — Trivy fails the pipeline on CRITICAL vulnerabilities, warns on HIGH.
  5. Unit & Integration Tests — pytest in-container; compose-based integration with health polling.
  6. Push — push to the container registry.
  7. Deploy → Staging → Smoke Test.
  8. Deploy → Production — gated by manual approval (rolling, zero-downtime).
  9. Verify — post-deploy health and rollout checks; rollback on failure.
Why a shared library? Instead of duplicating pipeline logic across six services, hardening the deploy process once means every service inherits the improvement instantly — consistency and a single point of change.

9. Infrastructure as Code (Terraform)

The entire AWS environment is described in Terraform, making it reproducible and peer-reviewable. The root module composes four child modules in dependency order.

module "networking"  { source = "./modules/networking" ... }   # VPC, subnets, SGs
module "eks_cluster" { source = "./modules/eks-cluster" ...
                       depends_on = [module.networking] }        # EKS control plane + nodes
module "rds"         { source = "./modules/rds" ...
                       depends_on = [module.networking] }        # Multi-AZ PostgreSQL
module "monitoring"  { source = "./modules/monitoring" ...
                       depends_on = [module.eks_cluster, module.rds] }
ModuleProvisions
networkingVPC, public/private/database subnets, NAT gateways, security groups
eks-clusterEKS control plane, managed node group, IAM roles, add-ons
rdsMulti-AZ PostgreSQL 15.4, KMS encryption with key rotation, automated backups, deletion protection
monitoringCloudWatch dashboards, alarms, SNS topics, log groups

Providers are pinned (AWS ~> 5.0, Kubernetes ~> 2.27); a commented S3 + DynamoDB backend is provided for remote state with locking. A secondary DR region (eu-west-1) is defined for cross-region backups and failover.

10. Monitoring & Observability

The observability stack is Prometheus, Grafana, and AlertManager. Prometheus scrapes every service's /metrics endpoint (15s default; emergency-dispatch at 5s as it is latency-critical) and supports both static targets (Docker Compose) and Kubernetes service discovery (EKS).

Team-aware routing ensures the correct responder is paged for each failure class — essential when seconds influence patient outcomes.

12. Disaster Recovery

Disaster recovery is a first-class deliverable. Recovery objectives are defined by service tier and backed by six operator runbooks plus scripted automation.

Figure 5 — Disaster-recovery map (runbooks ↔ scenarios ↔ RTO)

12.1 Recovery Objectives

TierServicesRTO Target
Tier 1flight-operations, emergency-dispatch, aircraft-comms5 minutes
Tier 2patient-records, medical-equipment15 minutes
Tier 3api-gateway, monitoring30 minutes

RPO for operational data is tight; Prometheus metrics tolerate up to one hour of loss (observability, not transactional data).

12.2 Runbooks

IDScenarioRecovery Mechanism
RB-001Service Pod FailureKubernetes self-heal; verify replica count
RB-002Pod CrashLoopBackOffDiagnose logs, roll back image
RB-003Aircraft Communication LossAlert flight-ops, restore comms within Tier-1 window
RB-004Database FailureRDS Multi-AZ auto-failover; integrity verification
RB-005Full Cluster FailureRebuild region from Terraform; restore from backup
RB-006Traffic Surge / Mass CasualtyHPA scales 2→10; absorb load automatically

Automation scripts: backup.sh (exports all K8s objects + RDS snapshot to S3), restore.sh, failover.sh (DR-region switch), and health-check-all.sh. A demo-full-recovery.sh script performs a live failure-injection and recovery walkthrough.

15. Conclusion & Future Work

AeroMed delivers a resilient, observable, and fully automated cloud-native platform purpose-built for mission-critical healthcare transportation. Six fault-isolated microservices run on Kubernetes with autoscaling and self-healing; end-to-end monitoring provides team-aware alerting; six tested disaster-recovery runbooks tie to tiered recovery objectives; Jenkins CI/CD on a shared library guarantees consistent delivery; and Terraform makes the entire environment reproducible.

Future Enhancements

  • Centralized logging — integrate the ELK/Loki stack for aggregated, searchable logs.
  • Secret management — migrate from Kubernetes Secrets to HashiCorp Vault with the Agent Injector.
  • Distributed tracing — add OpenTelemetry for cross-service request tracing.
  • Real persistence — replace mock data with live RDS-backed repositories.
  • Progressive delivery — canary/blue-green deployments via Argo Rollouts.
